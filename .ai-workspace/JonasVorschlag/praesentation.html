ADR-002 | Core Architecture

Shared Code

Heute

"UI-Library" (die Abhängigkeits-Library)

shared-ui mixt API Services, Komponenten, Views und Utils. Jede Änderung betrifft potenziell Client Office und Data Office gleichzeitig.

Neu

Ein Package: Auth + API Client

Ein privates npm-Package mit dem Code der in allen Apps identisch laufen muss. Versioniert und per npm update aktualisierbar.

Keine shared-ui mehr. UI-Komponenten kommen aus einer Registry, werden ins Projekt importiert und dort projektspezifisch angepasst. Keine gemeinsame Dependency, kein Breaking Change in anderen Apps.

ADR-003 | Core Architecture

Feature-basierte Struktur

src/
  app/ // Globaler Setup (Router, Providers)
  features/
    course-management/
      components/
      hooks/
      api.ts
      index.ts // Public API
  shared/ // App-weiter Code (z.B. Button.tsx)
Vorteile

Warum Feature-basiert?

  • Skalierbarkeit: Code wächst logisch mit den Features. Keine endlosen, unübersichtlichen "God-Folders" (`/components`, `/hooks`).
  • Isolation: Ein Feature ist eine geschlossene Kapsel. Alles (API, UI, State) lebt an einem Ort.
  • Strikte Boundaries: Features dürfen nicht quer aufeinander zugreifen. Kommunikation läuft ausschließlich über die `index.ts`.

ADR-005 | State & Data Flow

TanStack Router

Typsicherheit

100% Typsicherheit

URLs, Path-Params und Search-Params sind getypt. Falsche Links oder fehlende Parameter fallen beim Build auf, nicht erst im Browser.

Vorteile

Warum TanStack Router?

  • Search Params als State: Kritisch für komplexe Filter, Suchen und Pagination in der Lernwelt. State lebt in der URL, nicht lokal.
  • Nahtlose Integration: Gebaut vom selben Team wie TanStack Query. Cache-Prefetching beim Hovern von Links ist ein Einzeiler.

ADR-006 | State & Data Flow

State Management

Server State (~90%)

TanStack Query

Der Single Source of Truth für Backend-Daten (Kurse, Profile). Übernimmt Caching und Fetching vollständig.

  • Loading/Error States kommen out-of-the-box.
  • Automatisches Caching und Background-Refetching.
  • Globale Verfügbarkeit der Daten ohne Prop-Drilling.
Client State (~10%)

Zustand

Nur für flüchtigen UI-State (Sidebars, Theme). Weniger Boilerplate, mehr Code den wir löschen können.

  • Zero Boilerplate (im Gegensatz zu Redux).
  • Keine unnötigen Re-Renders.
  • Simples Speichern von Settings in den LocalStorage.

ADR-007 | State & Data Flow

React Hook Form + Zod

Vorteile

Warum React Hook Form + Zod?

  • Strikter Contract: Wir definieren ein einziges Zod-Schema, das gleichzeitig als Typescript-Type und als Formular-Validierung dient.
  • Performant: Uncontrolled Inputs by Default. Kein Re-Render bei jedem Tastendruck — relevant für große CMS-Formulare.
  • Ökosystem-Fit: Integriert sich nativ in TanStack Router (Search Validation) und shadcn/ui.

ADR-008 | UI Architecture

Tailwind CSS

Utility-First

Warum Utility-First?

  • Geschwindigkeit: Schnelles Styling direkt im Code. Kein ständiger Context-Switch zwischen TSX und CSS-Dateien.
  • Konsistenz: Entwickler erfinden keine eigenen Spacings/Farben mehr, sondern nutzen vordefinierte Token.
Colocation

Styles leben am Code

Styles leben direkt an der Komponente. Ungenutzte Klassen landen nicht im Build. Kein Suchen in separaten CSS-Dateien.

ADR-009 | UI Architecture

Kein CSS Preprocessor

Entscheidung

Warum kein SCSS/Less?

Strikte Regel: Kein SCSS, kein Less. Wir entfernen diese Build-Schritte komplett aus unserer Pipeline.

  • Weniger Komplexität: Eine Build-Dependency weniger. Keine Kollisionen zwischen SCSS-Syntax und Tailwind.
  • Natives CSS ist reif: Variablen (Custom Properties) und Nesting sind mittlerweile native CSS-Features. Preprocessors haben ihren Nutzen verloren.

ADR-010 | UI Architecture

Design Tokens & WhiteLabel

Theming

CSS Custom Properties

Jede App definiert ihre Farben, Fonts und Spacings in der globals.css. Komponenten greifen nur auf Token zu.

  • Eine Komponente (z.B. Button) greift auf `bg-primary` zu.
  • Das Token `primary` ist je nach App/Theme unterschiedlich definiert.
  • Gleiche Komponenten, unterschiedliches Look & Feel pro App — ohne Code zu duplizieren.
:root {
  --primary: #aa7939; /* Produkt A */
}

[data-theme="product-b"] {
  --primary: #3b82f6; /* Produkt B */
}
→

Globaler Theme-Switch via Token-Swapping

ADR-011 | UI Architecture

shadcn/ui

Konzept

Keine Blackbox

  • Wir besitzen den Code: Komponenten werden als Source-Code ins Projekt kopiert. Wir haben 100% Kontrolle über das Markup.
  • Barrierefreiheit (A11y): Basiert auf Radix UI (Fokus-Traps, Screenreader-Support, Keyboard-Nav). Wochenlange Arbeit wird uns geschenkt.
  • Flexibilität: Perfekte Symbiose mit ADR-010 (Design Tokens). Komponenten passen sich automatisch dem jeweiligen Produkt an.
AI-Tooling

shadcn ist breit in LLM-Training-Daten vertreten. Vercels v0 z.B. generiert direkt shadcn-Komponenten.

ADR-012 | DevEx & Tooling

Biome + Husky

Tooling

Blitzschnelles Tooling

  • Biome ersetzt ESLint + Prettier: Ein einziges Tool, das in Millisekunden lintet und formatiert (ca. 25x schneller).
  • Pre-Commit Hooks: Bevor ein Entwickler committen kann, wird der Code automatisch formatiert und geprüft.
Ergebnis

Formatting und Linting passieren automatisch. Kein Diskutieren über Whitespaces oder Semicolons im Code-Review.

ADR-013 | DevEx & Tooling

Testing Strategy

Unit

Vitest

Business-Logik, Utilities, Zod-Schemas. Schnell durch Vite-Integration.

Component

React Testing Library

Komponenten-Verhalten aus User-Sicht: Klicks, Formulare, Rendering — ohne echten Browser.

End-to-End

Playwright

Kritische User-Flows im echten Browser. Multi-Browser, automatisierbar in CI.

Ansatz

Kein Ziel von 100% Coverage. Tests sichern Refactorings ab. Ein fehlschlagender Test blockiert den Merge.
